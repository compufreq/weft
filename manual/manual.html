1 Getting started

Weft is a single container that runs next to your Weaviate instance. It needs no account, no configuration file, and no build step.

Requirements

  • Docker (or any OCI runtime). Images are published for amd64 and arm64.
  • A reachable Weaviate instance (1.30 or newer; the latest two minor versions are fully tested).

Run Weft

  1. Start the container, pointing it at your Weaviate: 
    docker run -d -p 8080:8080 \
  -e WEAVIATE_URL=http://your-weaviate:8080 \
  ghcr.io/compufreq/weft:1.4
  2. Open http://localhost:8080 in your browser.
  3. Your Weaviate appears on the Instances page, ready to browse. That is the whole setup.
Weft follows your operating system's dark/light preference. The theme selector in the navigation bar overrides it (System / Light / Dark); the choice is remembered by your browser.
If your Weaviate needs an API key, add -e WEAVIATE_API_KEY=your-key. The key stays inside the Weft container — your browser never talks to Weaviate directly.

Docker Compose

services:
  weaviate:
    image: cr.weaviate.io/semitechnologies/weaviate:1.37.2
    # ... your existing Weaviate config ...
  weft:
    image: ghcr.io/compufreq/weft:1.4
    environment:
      WEAVIATE_URL: http://weaviate:8080
    ports:
      - "8080:8080"

2 Managing instances

One Weft deployment can manage any number of Weaviate instances. The home page lists every registered instance.

The Instances page: registered instances on the left, the Add instance form on the right.

Add an instance at runtime

  1. On the home page, fill in the Add instance form: a display Name (e.g. “Staging cluster”) and the URL of the Weaviate HTTP API.
  2. If the instance requires authentication, paste its API key. The key is accepted once and never shown again.
  3. Click Add instance. A new card appears; click Browse schema → to start exploring it.

Remove an instance

  1. On the instance card, click Remove. This only deregisters the connection — nothing is deleted on the Weaviate side.
Do my instances survive a restart? Instances added in the UI are kept in memory. To persist them, start Weft with WEFT_INSTANCES_FILE=/data/instances.json and mount a volume at /data — every add/remove is then saved automatically. Fleet-style permanent configs belong in weft.yaml (see chapter 15).

3 Authentication & read-only mode

By default Weft is open — anyone who can reach the port can use it. Two deployment switches lock it down.

Protect the deployment with a token

  1. Generate a strong token and start Weft with it: 
    docker run -d -p 8080:8080 \
  -e WEAVIATE_URL=http://weaviate:8080 \
  -e WEFT_AUTH_TOKEN=$(openssl rand -hex 24) \
  ghcr.io/compufreq/weft:1.4
  2. Visiting Weft now shows the token prompt. Enter the token and click Unlock.
  3. Your browser receives a secure session cookie valid for 7 days. A Log out button appears in the navigation bar; clicking it ends the session immediately.

The token gate. Wrong tokens are rejected, and more than 5 attempts per minute are throttled with a countdown.

API clients (scripts, curl) authenticate with a header instead: Authorization: Bearer <token>. Health probes (/healthz, /readyz) always stay open for orchestrators.

Read-only mode

  1. Start Weft with -e WEFT_READ_ONLY=true.
  2. An amber banner appears on every page, and every write control in the UI disappears — no New collection, no editing, no imports, no backups.
  3. Browsing, filtering, search, aggregations, the GraphQL console, and exports keep working normally.

Read-only mode: the banner is visible and all write controls are hidden. The server also rejects any mutation, so the protection does not depend on the UI.

Combine both switches to give a whole team safe visibility: a read-only deployment with a shared token.

4 Browsing schemas

Click Browse schema → on any instance card to see its collections.

The Schema page: every collection with its vectorizer, index type and multi-tenancy status, plus the alias panel at the bottom.

  1. The table lists every collection. Click a collection name for its detail page.
  2. Export JSON downloads the complete schema as a file — useful for backups and for diffing later.
  3. The buttons at the top right lead to Compare… (chapter 14), Ops (chapter 13) and the Console (chapter 11).

Collection detail

A collection's detail page: vectorizer, index type, the full property table, and management actions.

  1. The summary cards show the vectorizer, vector index type and property count.
  2. The Properties table lists every property with its data type.
  3. Expand Raw definition (JSON) to inspect the exact class definition Weaviate stores.
  4. Browse objects → jumps to the data explorer for this collection (chapter 7).

5 Creating & managing collections

Create a collection

The guided New collection form. The raw JSON toggle accepts a full class definition for advanced setups.

  1. On the Schema page, click New collection.
  2. Enter a Name — UpperCamelCase, e.g. Product.
  3. Tick Multi-tenant if each of your customers/tenants should get isolated data (chapter 12).
  4. Add properties: a name plus a type (text, int, number, boolean, date, text[], uuid). Use + Add property for more rows.
  5. Click Create collection. It appears in the schema table immediately.
The form creates collections with vectorizer: none (bring your own vectors). To configure a vectorizer module, tick raw JSON and paste a complete class definition instead.

Add a property to an existing collection

  1. Open the collection's detail page.
  2. Below the property table, enter the Property name and pick a Type.
  3. Click Add property. (Adding is the only in-place schema change Weaviate supports — renaming or retyping a property requires a migration to a new collection.)

Delete a collection

Deletion is armed only after typing the exact collection name.

  1. On the detail page, click Delete collection….
  2. Type the collection's exact name into the confirmation box — the red button stays disabled until it matches.
  3. Click Delete forever.
Deleting a collection permanently removes every object in it on the Weaviate side. There is no undo. Consider a backup first (chapter 13).

6 Collection aliases

Aliases (Weaviate ≥ 1.32) let applications query a stable name like ArticlesLive while you swap the underlying collection — the standard pattern for zero-downtime reindexing and migrations.

The Aliases panel sits at the bottom of the Schema page (visible in the chapter 4 screenshot).

Create an alias

  1. In the Aliases panel, type the New alias name.
  2. Choose the collection it Points at.
  3. Click Create alias.

Repoint an alias (migration cut-over)

  1. Build and fill your new collection (e.g. ArticleV2).
  2. In the alias row, change the target dropdown from ArticleV1 to ArticleV2. The switch is instant for every client using the alias.

Delete an alias

  1. Click the × on the alias row. Only the alias is removed — the target collection and its data stay untouched.
On Weaviate older than 1.32 the panel simply explains that aliases are not available — nothing breaks.

7 Exploring objects: browse, filter & aggregate

From a collection's detail page, click Browse objects →. The explorer has four tabs: Browse, Search, Map and Import.

Browse: the object table with live aggregations above it and the JSON detail panel on the right.

Browse and inspect

  1. Objects load 25 at a time; click Load more to append the next page (cursor-based, fast at any collection size).
  2. Click any row to open the full object as JSON in the right-hand panel.
  3. Export NDJSON (top right) streams the entire collection to a file, one JSON object per line — constant memory, any size.

Filter

A category filter applied: the table, the match count and the facet bars all reflect it.

  1. Click + Add filter. Pick a property, an operator and a value. Operators adapt to the property type — text offers Equal/Like/ContainsAny, numbers offer comparisons, and IsNull finds missing values.
  2. Add more rows to combine conditions. With two or more, a match mode select appears: Match all (AND) or Match any (OR).
  3. Click + Add group for nested logic — e.g. wordCount < 100 and (category = science or category = sports). Each group has its own match mode; a new group starts in the opposite mode of its parent, which is usually what you want. Groups nest up to three levels.
  4. Click Apply filters to run. Like supports wildcards: *science* matches anywhere in the text. ContainsAny takes a comma-separated list.
  5. Clear removes all filters and groups.

Aggregate

  1. The Aggregations card always shows the count of objects matching the current filters.
  2. Choose a property under Facet by to see value distribution as sorted bars (e.g. objects per category). Facets respect the active filters.

8 Searching: keyword, vector & hybrid

A BM25 search with ranked, scored results. The active filter from the filter bar constrains the search too.

  1. Switch to the Search tab and pick a Mode:
    ModeWhat it doesNeeds
    BM25Classic keyword relevancenothing extra
    HybridFuses keyword + vector rankingoptional vector
    nearVectorSimilarity to a raw vector you pastea JSON vector, e.g. [0.1, 0.2, …]
    nearTextSemantic search from plain texta vectorizer module on the collection
  2. Type your query (or paste a vector) and click Search.
  3. Results are ranked with their score (BM25/hybrid) or distance (vector). Click a hit to inspect its full JSON.
  4. Filters set on the filter bar apply to searches as well — e.g. search “demo” only within category = science.
If nearText reports that no vectorizer is configured, the collection stores raw vectors only — use nearVector or hybrid instead.

9 The vector map

The Map tab: every object vector projected to 2D. Nearby points are semantically similar; colors follow the most facet-like text property.

  1. Open the Map tab. Weft loads up to 200 objects with their vectors and projects them to two dimensions (PCA) directly in your browser.
  2. Points that sit close together are similar in vector space — clusters usually mean coherent topics; outliers are worth inspecting.
  3. Hover a point for its label; click it to load the full object into the detail panel.
  4. The legend shows the automatic coloring — Weft picks the text property with the fewest distinct values (e.g. category).
If the collection stores no vectors (objects inserted without them and no vectorizer), the map explains that instead of showing an empty plot.

10 Editing data: create, edit, delete & import

All of this chapter is unavailable in read-only deployments — the controls are hidden and the server rejects the operations.

Create one object

  1. In the explorer, click New object (top right).
  2. Enter the properties as a JSON object, e.g. { "title": "Hello", "wordCount": 42 }.
  3. Click Save. The list refreshes with your object included.

Edit an object

The editor pre-filled with the selected object's properties.

  1. Select an object in the table, then click Edit in the detail panel header.
  2. Change the JSON and click Save. This is a full replacement of the object's properties (Weaviate PUT semantics).

Delete an object

  1. Select the object, click Delete in the detail panel header, and confirm. This cannot be undone.

Batch import

Importing NDJSON: two lines inserted, with a per-item error report when something is rejected.

  1. Open the Import tab.
  2. Paste a JSON array of objects, NDJSON (one JSON object per line), or CSV with a header row — or load any of these from a file with the file picker. Weft's own NDJSON export re-imports directly, so export → import moves data between collections or instances.
  3. For CSV, the header names the properties and values are typed from the collection schema (int, number and boolean columns are converted; quoted fields with commas work as usual). Two optional special columns: id sets the object UUID and vector takes a quoted JSON array like "[0.1, 0.2]". Empty cells are simply omitted. Parsing happens entirely in your browser.
  4. Click Import. Objects are inserted in batches (up to 10,000 per import).
  5. The report shows how many were inserted and failed, with Weaviate's exact error and the line number for each failure — one bad row never aborts the rest.

11 The GraphQL console

The console: query on the left, pretty-printed response on the right.

For anything the UI doesn't cover — nested filter logic, Explore queries, custom _additional fields — use the raw console.

  1. From the Schema page, click Console.
  2. Write any Weaviate GraphQL query (Get, Aggregate, Explore). Press Ctrl+Enter or click Run.
  3. The full response envelope renders on the right — including Weaviate's own error messages, exactly as sent.
  4. Your query buffer is saved in the browser per instance, so it survives reloads.
The console is always safe: Weaviate's GraphQL API is query-only (all writes go over REST), so the console works even in read-only deployments.

12 Multi-tenancy

The Tenants tab: activity status and per-tenant object counts, with activate/deactivate actions.

Multi-tenant collections isolate each tenant's data in its own shard. Weft treats tenants as first-class citizens.

  1. On a multi-tenant collection (badge on its detail page), open the Tenants tab.
  2. The table lists every tenant, its status (HOT = active/queryable, COLD = deactivated, frees memory) and its object count.
  3. Create tenants: enter one or more names and confirm.
  4. Deactivate a tenant to release resources; Activate restores it. Data is preserved either way.
  5. Click Browse on a HOT tenant to open the object explorer scoped to that tenant — the tenant field is pre-filled, and search/filters/import all respect it.
When browsing a multi-tenant collection you must set the tenant field (top right of the explorer) — Weaviate requires it for every data operation on such collections.

13 Operations: cluster health, backups & RBAC

The Ops page (top): node health cards, the shard table and capabilities.

The Ops page (bottom): backups, Raft cluster statistics and the access-control panel.

Node & shard health

  1. From the Schema page, click Ops. Node cards show status, version and totals; the shard table breaks down objects per shard (including per-tenant shards). The view refreshes every 10 seconds.

Backups

  1. The Backups section appears when the instance has a backup module enabled (backup-filesystem, backup-s3, backup-gcs or backup-azure).
  2. Click Create backup to snapshot all collections to the configured backend; the list shows each backup's status (STARTED → SUCCESS).
  3. Restore brings a backup's collections back. Weaviate refuses to overwrite collections that still exist — restore is for recovering deleted data or seeding a fresh instance.

Live metrics (v1.2+)

When Weaviate runs with PROMETHEUS_MONITORING_ENABLED=true, the Ops page adds live charts: heap in use, goroutines, CPU cores, requests per second, object counts (total and per collection) and vector-index size. Weft scrapes <weaviate-host>:2112/metrics by default (configurable per instance via metrics_url) and keeps a rolling ~5-minute window in your browser — no metrics database involved. If the endpoint is unreachable the panel says so and everything else keeps working.

Cluster statistics

The statistics card shows the Raft consensus state: whether the cluster is synchronized, which node is the leader, and each node's health — the first thing to check when a multi-node cluster misbehaves.

Access control (RBAC)

If the instance runs with RBAC enabled and Weft's key may read it, this panel lists every role (with its permission count) and every user with their assigned roles. On instances without RBAC the panel says so honestly — nothing to configure.

Managing RBAC (v1.3+): with a privileged key (and outside read-only mode) the panel is editable: create a role with a guided permission (action + collection filter), click a role to expand its permissions and add/remove them, delete a role behind a confirmation, and assign or revoke roles on any user with the inline controls. All mutations are enforced server-side.

14 Comparing schemas (diff)

The diff page: compare this instance's schema against another instance or a pasted export.

  1. From the Schema page, click Compare….
  2. Choose the comparison target: another registered instance (drift between staging and production) or pasted JSON (an earlier Export JSON — drift over time).
  3. The result lists every difference: added/removed collections, added/removed properties, and changed fields with their left/right values.

weft.yaml — permanent multi-instance config

Mount a weft.yaml at /app/weft.yaml to define permanent instances:

listen: "0.0.0.0:8080"
instances:
  - id: local
    name: Local Weaviate
    url: http://weaviate:8080
  - id: staging
    name: Staging cluster
    url: https://weaviate.staging.example.com
    api_key: "sk-..."
API keys in weft.yaml and in the instances file are stored in plain text — protect those volumes. Keys are never shown in the UI or API responses.

Upgrades

Weft is stateless: upgrading is docker pull + recreate. Your Weaviate data is never touched. The /api/v1 HTTP API is stable across all 1.x releases (additive changes only). Pin at least the minor tag (ghcr.io/compufreq/weft:1.4) in production.

16 Troubleshooting

SymptomLikely cause & fix
“Could not load schema” / 502 errorsWeft can't reach the Weaviate URL. From inside Docker, localhost refers to the Weft container itself — use the service name (http://weaviate:8080) or host.docker.internal.
Browsing a collection returns an error about tenantsThe collection is multi-tenant: set the tenant field (top right of the explorer) first.
nearText returns an errorThe collection has no vectorizer module. Use BM25, hybrid or nearVector instead.
“Too many attempts — try again in …s”The login rate limit (5/minute per IP). Wait for the countdown, then retry with the correct token.
Buttons like New object / Import are missingThe deployment runs in read-only mode, or your session is on a read-only instance of Weft. This is by design.
Aliases panel says they need Weaviate ≥ 1.32Your Weaviate is older. Upgrade Weaviate, or ignore — everything else works.
Import reports failuresRead the per-item message: usually a property value that doesn't match the schema type (e.g. text where an int is expected). Fix those lines and re-import them.
Instances added in the UI vanish after restartPersistence isn't enabled. Set WEFT_INSTANCES_FILE with a volume (chapter 2), or define instances in weft.yaml.
Vector map says no vectors foundObjects were inserted without vectors and the collection has no vectorizer. The map needs stored vectors.
